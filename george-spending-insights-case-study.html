Simplifying Everyday Banking —
Rethinking Spending Insights in George

A concept exploration into how digital banking could help users understand their finances through clearer structure, visual hierarchy, and simplified interactions.

The data was never the problem. The structure around it was.

<\!-- PROBLEM -->
01 — Problem

The challenge with financial data

Managing personal finances is often unclear and fragmented. Users are presented with long lists of transactions but lack meaningful context to make sense of them. Without clear patterns or visual signals, it becomes difficult to understand spending behavior or make confident decisions.

"I know my balance, but I don't know where my money actually goes every month."

🔍 What users see today

Chronological lists of transactions with merchant names, amounts, and dates — but no layer of meaning on top of them. No patterns, no groupings, no behavioral context.

💡 What users actually need

A quick, clear answer to "how am I doing this month?" — without having to mentally tally up numbers or switch between multiple screens to form a picture.

<\!-- CONTEXT + GOALS -->
02 — Context & Goals

Designing for everyday users

This concept targets everyday banking users — not financial power users. They want a quick, actionable snapshot of their finances without navigating complex tools or dashboards. The design should meet them where they are: briefly checking in on their phone between other activities.

Design Goals

1

Make spending understandable at a glance

Users should form a clear picture of their month within seconds of opening the insights view — no mental math required.

2

Reduce cognitive load in financial decision-making

Strip away noise. Elevate the information that matters. Let visual hierarchy do the heavy lifting so the user's brain doesn't have to.

3

Create a consistent, intuitive experience across flows

Build with a small set of reusable components so the experience feels coherent from home screen through to category detail.

04 — Wireframes

Structure before visual design

Before committing to visual decisions, the work began with wireframes focused purely on information hierarchy, content priority, and interaction flow. These low-fidelity layouts established the skeleton that later visual polish would be applied to.

The key question at this stage: what information matters most, and where should the user's eye go first?

<\!-- WF 1 -->
Screen 1 — Home / Overview
Key decisions: Balance takes visual priority. Insights entry point positioned prominently above the transaction list — not buried.
<\!-- WF 2 -->
Screen 2 — Spending Insights (Hero)
Key decisions: Chart gives visual orientation first. Category rows provide specifics below. Hierarchy flows naturally top to bottom.
<\!-- WF 3 -->
Screen 3 — Category Detail
Key decisions: Micro-insights (avg spend, frequency) surfaced above the list — they give context before details.
<\!-- WF 4 -->
Screen 4 — Action Layer
Key decisions: Suggestions over controls. Simple toggles, not complex inputs. Avoids budgeting complexity.
04.5 — Before vs After

Same data, different structure

Before
Dense, hard to scan

Pie chart, bar chart, goal tracker, and a data table all visible at once. No clear starting point — users had to read everything to understand anything.

After
Structured, scannable in seconds

One chart for trend, one number for the total, a ranked list for categories. Users get orientation first — detail only when they want it.

05 — Final Screens

The designed experience

With wireframes established, visual design was applied with a focus on calm, structured hierarchy. The palette is restrained — near-whites, deep blues for primary actions, and subtle category colors. Typography weight does the work that color might otherwise overdo.

<\!-- ITERATION -->
06 — Iteration

How the design evolved

Three versions, each fixing a specific failure from the one before.

Version 1
Dense Dashboard

Showed a pie chart, bar chart, goal tracker, savings widget, and a data table on one screen. Users had no entry point — every element demanded equal attention, so they read none of it.

Removed
Pie chart Goal tracker Savings widget Data table
Problem

No hierarchy meant no starting point. Users scanned the whole screen and left with nothing retained.

Version 2
Card-per-Category

Dropped all charts, replaced with one card per spending category. Less overwhelming, but removing the chart also removed any sense of proportion — users couldn't tell at a glance which category was the biggest.

Changed
Charts removed Category cards Monthly total visible
Problem

Text-only layout forced users to read and compare numbers. No visual shortcut.

Version 3 · Final
Chart + Ranked List

Added a single bar chart at the top for visual orientation, then a ranked category list with inline proportion bars below. Users scan the chart first, then read only what they care about.

Added
Bar chart (trend) Inline proportion bars Month-on-month delta
Why it works

Chart answers "how is this month overall?" in one glance. The list answers "where did it go?" without an extra tap.

<\!-- SYSTEM -->
07 — Design System Thinking

Built to extend, not just to ship

  • 🧩
    Components built once, used everywhere. The category row, insight module, and transaction row each appear across multiple screens without modification. Designing them once meant every screen that uses them is automatically consistent.
  • 📐
    The same hierarchy rules apply on every screen. Heading sets context, a key number anchors the view, supporting detail sits below. Users only need to learn the pattern once.
  • 📦
    Adding a new category or screen requires no new patterns. A new spending category just gets a color dot and slots into the existing row component. A new screen composes from the same building blocks.
  • 🎨
    A constrained token set keeps visual noise low. One primary blue, four category colors, two surfaces, three type sizes. Fewer choices per decision means faster design and fewer inconsistencies.

Tokens applied across all screens

One primary blue, four category colors, two background surfaces, three type sizes.

<\!-- OUTCOME -->
08 — Outcome

What this concept achieves

This concept simplifies how users interact with financial data by removing complexity and focusing on clarity. The result is a more intuitive experience that supports faster understanding and more confident decision-making — without adding cognitive overhead.

⚡

Faster financial orientation

Users can understand their monthly spending picture in seconds — without tallying up individual transactions.

🧠

Reduced cognitive load

Visual hierarchy and restraint mean the interface does the analytical work, freeing users to focus on decisions rather than data processing.

🔄

Scalable component system

The component library is small but composable — future features can be introduced without breaking visual coherence.

<\!-- REFLECTION -->

Designer's Reflection

This case study bridges my background in industrial design with product UX thinking. Industrial design taught me that the best solutions are often the most restrained ones — that removing material is as important as adding it. That principle translated directly here.

From industrial → digital

In physical product design, every element must earn its place — material, weight, and form are constrained resources. The same discipline applied to this interface: every component was questioned before it was included.

What I would explore next

User testing with George users to validate whether the chart-first approach aids comprehension. Accessibility review of color usage for category differentiation. And a deep dive into the notification/awareness system.
